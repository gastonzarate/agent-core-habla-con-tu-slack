Cómo funciona · base vectorial

Los mensajes se agrupan por significado

Cada mensaje es un punto en el espacio. Una pregunta cae cerca de los mensajes parecidos → buscar = encontrar los vecinos más cercanos.

Acá explicás el concepto: cada mensaje se convierte en un vector y queda ubicado en este espacio. Los que hablan de lo mismo (deploys, bugs, clientes) se agrupan solos. Cuando preguntás, tu pregunta también se vuelve un punto y cae cerca de su grupo: el retrieval es literalmente "traeme los puntos más cercanos" (las líneas blancas). No busca palabras, busca cercanía de significado. Mové el mouse para rotar.
Knowledge Base · qué hace por dentro

Vos le das el documento. Él hace el resto.

Le entregás tus documentos (acá, cada mensaje). El Knowledge Base los parte si son largos, los vuelve embeddings y los indexa. Todo solo.

Versión visual del paso anterior. Le entregás tus documentos (en este caso, cada mensaje de Slack es un documento). El Knowledge Base los parte en chunks si son largos, genera los embeddings con Titan (los vectores del espacio 3D) y los indexa en S3 Vectors. Recalcá: todo eso lo hace AWS solo; nosotros solo entregamos los documentos. Aclará que como los mensajes son cortos, muchas veces cada mensaje ya es un chunk — el chunking pesa en documentos largos.

Este es el corazón de un agente: el loop. Mirá el transcript a la derecha — el agente piensa, decide llamar ask_kb, observa el resultado, y como le falta info (el freeze) razona y llama la tool OTRA VEZ. Recién cuando tiene todo, responde. La etapa activa se ilumina a la izquierda. El punto: el agente decide qué herramientas usar y cuántas iteraciones hacer — eso es lo que lo diferencia de una sola llamada a un modelo.

Amazon Bedrock AgentCore · los bloques

Qué le podés sumar al agente

Servicios gestionados que se enchufan al agente sin reescribirlo — usás solo los que necesitás.

🚀 Runtime Hosting serverless, sesión aislada por usuario. Escala solo.
🧠 Memory Memoria de corto y largo plazo entre conversaciones.
🔌 Gateway Convierte APIs y Lambdas en tools (MCP) sin código pegamento.
🔐 Identity Acceso seguro a recursos en nombre del usuario (OAuth).
📈 Observability Trazas, métricas y debugging del agente (OpenTelemetry).
🧮 Code Interpreter Ejecuta código en un sandbox aislado, a pedido del agente.
🌐 Browser Navegación web headless y segura para tareas en la web.
🛡️ Guardrails Filtra temas y datos sensibles, en entrada y salida.
AgentCore es una caja de herramientas modular: cada bloque es un servicio gestionado que se enchufa al agente sin reescribir nada. Runtime lo hostea, Memory le da memoria, Guardrails lo protege (los tres resaltados, los que potencian este caso). ¿Conectar la API interna del cliente como tools? Gateway. ¿Que actúe en nombre del usuario con sus permisos? Identity. ¿Visibilidad y gobernanza? Observability. ¿Ejecutar código o navegar la web? Code Interpreter y Browser. Mensaje de negocio: empezás mínimo y escalás capacidades a medida que el caso crece — sin rehacer la arquitectura.
🧠 AgentCore Memory · corto plazo

Que recuerde la conversación

El historial de la charla, guardado por sesión. En cada turno el agente repite este ciclo — sin que el código lo maneje:

📥Cargar últimos turnos de la sesión
→
💬Responder con ese contexto
→
💾Guardar el turno nuevo
↺ se repite en cada mensaje
¿qué se decidió del deploy?
El deploy de prod quedó para el viernes 18hs.
¿y eso cuándo era?
Te referís al deploy: quedó para el viernes 18hs. ✅
Sin memoria, "eso" no significa nada. Con corto plazo, el agente sigue el hilo.

También hay largo plazo: extrae hechos y preferencias que persisten entre sesiones.

El agente sin memoria arranca de cero cada vez. Corto plazo = el historial de la charla por sesión: lo cargamos antes de responder y guardamos el turno nuevo (el ciclo de la izquierda). Por eso, mirá el chat de la derecha, podés decir "y eso cuándo era" y entiende que hablás del deploy. Y hay largo plazo si querés que recuerde cosas entre conversaciones distintas.

Lo que cuesta construirlo

Un agente, en ~15 líneas

app = BedrockAgentCoreApp()

@tool
def ask_kb(pregunta):
    "Busca en el conocimiento del equipo."
    return buscar_en_knowledge_base(pregunta)

agente = Agent(model="claude-sonnet-4-6", tools=[ask_kb])

@app.entrypoint
def invoke(payload):
    return agente(payload["prompt"])

Esto es casi todo el agente. El hosting, la memoria, la seguridad y la observabilidad los pone AgentCore — no se programan.

No hace falta que lean el código. El punto visual: ESTO es prácticamente todo lo que se escribe. Definís una herramienta (buscar en el conocimiento), creás el agente con un modelo, y un punto de entrada. Todo lo demás —escalar, memoria, seguridad, métricas— lo pone AgentCore. Por eso un equipo chico puede tener esto en producción en días, no meses. Ese es el cambio.
