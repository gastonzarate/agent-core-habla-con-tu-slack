Hands-on · Paso 1

La base vectorial (S3 Vectors)

Primero, dónde van a vivir los embeddings. Creamos:

  • un vector bucket → el almacén
  • un índice → la búsqueda por similitud

Dos parámetros que importan:

  • dimension 1024 → el tamaño del vector de Titan v2
  • cosine → cómo se mide "parecido"

Serverless: se paga por uso, sin infra.

s3v.create_vector_bucket(
    vectorBucketName="slackrag-vectors")

s3v.create_index(
    vectorBucketName="slackrag-vectors",
    indexName="slackrag-index",
    dataType="float32",
    dimension=1024,          # = Titan v2
    distanceMetric="cosine",
)
python workshop/steps/step1_vectors.py
Analogía: una biblioteca vacía con las estanterías ya organizadas por tema (el espacio vectorial). Todavía no hay libros. dimension 1024 DEBE coincidir con el modelo de embeddings. Correr el step en vivo (idempotente).

Crear el Knowledge Base

Una data source CUSTOM nos deja inyectar texto directo por API — sin bucket intermedio.

Hands-on · Paso 4

El agente que razona

Antes hacíamos retrieve a mano. Ahora un agente (Strands):

  • recibe la pregunta
  • decide usar la tool ask_kb
  • le pasa el contexto a Claude
  • Claude redacta la respuesta + citas

Dos tools: ask_kb y ingest_slack.
Por ahora corre local — sin desplegar nada.

@tool
def ask_kb(question: str) -> str:
    """Responde sobre el Slack indexado, con citas."""
    ctx, cites = retrieve(kb_id, question)
    return f"{ctx}\n\n[Fuentes: {cites}]"

agent = Agent(
    model=BedrockModel("claude-sonnet-4-6"),
    tools=[ask_kb, ingest_slack],
    system_prompt="Respondé sobre el Slack del equipo, "
                  "siempre con las fuentes.",
)

agent("¿cuándo es el deploy de prod?")
python workshop/steps/step4_agent.py
Line-highlights: (1) la tool ask_kb es solo una función decorada; (2) el agente = modelo + tools + prompt; (3) lo invocás como una función. Correr en vivo y mostrar el "Tool #1: ask_kb" (el agente DECIDIENDO usar la herramienta) → ese es el momento "agente", no solo RAG.
Hands-on · Paso 5

Llevarlo a producción

El agente local pasa a AgentCore Runtime:

  • hosting serverless, sesión aislada
  • observabilidad incluida
  • sin contenedores ni servidores

Con el CLI agentcore: configure arma la config, deploy publica.

direct_code_deploy → no necesita Docker local.

# 1) configurar el runtime
agentcore configure \
    -e agent/agent.py \
    -er $RUNTIME_ROLE

# 2) desplegar con sus variables
agentcore deploy \
    --env KB_ID=$KB_ID \
    --env MODEL_ID=claude-sonnet-4-6
python workshop/steps/step5_deploy.py --run
El salto: de "corre en mi máquina" a "serverless en AWS". Recalcar direct_code_deploy (sin Docker) y que el deploy tarda 1-2 min empaquetando deps. Tras desplegar: agentcore invoke '{"prompt":"..."}' para probar. Acá se puede mostrar la consola de Observability con la traza del agente (momento wow).
Hands-on · Paso 6

Conectar Slack

Slack no puede llamar al agente directo: manda un webhook y exige responder en < 3s.

Una Lambda‑puente fina:

  • valida la firma del request
  • ackea al instante
  • invoca al agente async
  • postea la respuesta en el hilo
def lambda_handler(event, ctx):
    if not firma_valida(event):        # viene de Slack?
        return {"statusCode": 401}

    # ack inmediato (< 3s) + trabajo async
    self_invoke_async(event)
    return {"statusCode": 200, "body": "🔎 Buscando..."}

# (async) → invoca AgentCore y postea en el hilo
answer = invoke_agent_runtime(prompt)
chat_postMessage(channel, answer, thread_ts)
python workshop/steps/step6_slack.py
El "porqué Lambda": el límite de 3s de Slack + validación de firma + invocación async. El step empaqueta bridge/ (autocontenido) y crea Lambda + API Gateway. Devuelve la Request URL → pegarla en la Slack App (Event Subscriptions + slash) y reinstalar. Mostrar la @mención real.
Hands-on · Paso 7

Que se actualice solo

Una regla de EventBridge dispara la ingesta cada 30 minutos.

  • lee los canales donde está el bot
  • normaliza e ingesta al KB

Idempotente: el id es canal‑ts, así que re‑ingestar sobrescribe, no duplica. Sin llevar estado.

events.put_rule(
    Name="slackrag-ingest-30min",
    ScheduleExpression="rate(30 minutes)",
)

events.put_targets(Rule="slackrag-ingest-30min",
    Targets=[{"Arn": bridge_lambda,
              "Input": '{"__scheduled_ingest__": true}'}])
python workshop/steps/step7_schedule.py
Cierre del hands-on: el sistema ahora se alimenta solo. Recalcar la idempotencia (id canal-ts → upsert) que evita duplicados sin estado. Tras este paso: el bot responde sobre datos reales que se refrescan cada 30 min. Demo final: @mención en Slack.
