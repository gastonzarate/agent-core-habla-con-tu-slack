Hands-on · Paso 1

La base vectorial (S3 Vectors)

Primero, dónde van a vivir los embeddings. Creamos:

  • un vector bucket → el almacén
  • un índice → la búsqueda por similitud

Dos parámetros que importan:

  • dimension 1024 → el tamaño del vector de Titan v2
  • cosine → cómo se mide "parecido"

Serverless: se paga por uso, sin infra.

s3v.create_vector_bucket(
    vectorBucketName="slackrag-vectors")

s3v.create_index(
    vectorBucketName="slackrag-vectors",
    indexName="slackrag-index",
    dataType="float32",
    dimension=1024,          # = Titan v2
    distanceMetric="cosine",
)
python -m s1_vector_bucket.main
Analogía: una biblioteca vacía con las estanterías ya organizadas por tema (el espacio vectorial). Todavía no hay libros. dimension 1024 DEBE coincidir con el modelo de embeddings. Correr el step en vivo (idempotente).
Cómo funciona · base vectorial

Los mensajes se agrupan por significado

Cada mensaje es un punto en el espacio. Una pregunta cae cerca de los mensajes parecidos → buscar = encontrar los vecinos más cercanos.

Acá explicás el concepto: cada mensaje se convierte en un vector y queda ubicado en este espacio. Los que hablan de lo mismo (deploys, bugs, clientes) se agrupan solos. Cuando preguntás, tu pregunta también se vuelve un punto y cae cerca de su grupo: el retrieval es literalmente "traeme los puntos más cercanos" (las líneas blancas). No busca palabras, busca cercanía de significado. Mové el mouse para rotar.
Calidad de datos · garbage in, garbage out

Cómo embebés importa

Los mismos mensajes, crudos vs preprocesados (limpiar + resumir antes de embeber). Mirá cómo la pregunta pasa de traer ruido a caer en su grupo.

Punto clave que casi nadie cuenta: la calidad del RAG depende de QUÉ embebés. Si metés los mensajes crudos —joins, emojis, hilos mezclados, links— los vectores quedan dispersos y la búsqueda trae ruido (panel izquierdo). Lo correcto es preprocesar: limpiar y resumir cada mensaje/hilo, y embeber ESE resumen (panel derecho) → clusters limpios, retrieval preciso. En el demo de hoy lo mantuvimos simple (embebemos el texto), pero en producción este paso de preprocesamiento es el que más mueve la aguja en calidad. Es una de las "variantes".
Hands-on · Paso 2

Crear el Knowledge Base

El KB es RAG gestionado. Le decís dónde guardar y con qué modelo, y él se encarga de:

  • chunking de los documentos
  • embeddings con Titan v2
  • indexar en S3 Vectors (paso 1)

Una data source CUSTOM nos deja inyectar texto directo por API — sin bucket intermedio.

ba.create_knowledge_base(
    name="slackrag-kb",
    roleArn=role_arn,
    knowledgeBaseConfiguration={
      "type": "VECTOR",
      "vectorKnowledgeBaseConfiguration": {
        "embeddingModelArn": titan_v2_arn,   # 1024 dims
      }},
    storageConfiguration={
      "type": "S3_VECTORS",                  # 👈 serverless
      "s3VectorsConfiguration": {
        "vectorBucketArn": bucket, "indexArn": index}},
)
python -m s2_knowledge_base.main
Recorrer el código con los line-highlights: (1) nombre+role, (2) el modelo de embeddings, (3) storage = S3_VECTORS, el punto serverless. Después correr el step EN VIVO: es idempotente, si ya existe lo reusa. Mostrar el output con knowledgeBaseId.
Knowledge Base · qué hace por dentro

Vos le das el documento. Él hace el resto.

Le entregás tus documentos (acá, cada mensaje). El Knowledge Base los parte si son largos, los vuelve embeddings y los indexa. Todo solo.

Versión visual del paso anterior. Le entregás tus documentos (en este caso, cada mensaje de Slack es un documento). El Knowledge Base los parte en chunks si son largos, genera los embeddings con Titan (los vectores del espacio 3D) y los indexa en S3 Vectors. Recalcá: todo eso lo hace AWS solo; nosotros solo entregamos los documentos. Aclará que como los mensajes son cortos, muchas veces cada mensaje ya es un chunk — el chunking pesa en documentos largos.

Un buen prompt, por dentro

🧠 esto es lo que recibe el modelo

🎯 ObjetivoContestar la consulta de forma precisa y accionable.
📋 InstruccionesRespondé en español, claro y breve. Usá solo el contexto.

💡 EjemplosP: «¿cuándo es el deploy?» → R: «Viernes 18hs 📎[1]»

Mostrá que el prompt real es esto: un bloque con partes. Recorré de arriba a abajo — Rol, Objetivo, Instrucciones, y lo nuevo de los agentes (Herramientas y Contexto recuperado por RAG), Ejemplos (few-shot), Formato y Guardrails. Lo último, el Usuario, es lo ÚNICO que la persona escribe. Todo lo de arriba lo arma el agente por detrás. Esto es literalmente el system_prompt + tools + contexto que vemos en el código.

Este es el corazón de un agente: el loop. Mirá el transcript a la derecha — el agente piensa, decide llamar ask_kb, observa el resultado, y como le falta info (el freeze) razona y llama la tool OTRA VEZ. Recién cuando tiene todo, responde. La etapa activa se ilumina a la izquierda. El punto: el agente decide qué herramientas usar y cuántas iteraciones hacer — eso es lo que lo diferencia de una sola llamada a un modelo.
Amazon Bedrock AgentCore · los bloques

Qué le podés sumar al agente

Servicios gestionados que se enchufan al agente sin reescribirlo — usás solo los que necesitás.

USADO 🚀 Runtime Hosting serverless, sesión aislada por usuario. El paso 5.
USADO 🧠 Memory Memoria de corto y largo plazo entre conversaciones. El paso 4.1.
🔌 Gateway Convierte APIs y Lambdas en tools (MCP) sin código pegamento.
🔐 Identity Acceso seguro a recursos en nombre del usuario (OAuth).
📈 Observability Trazas, métricas y debugging del agente (OpenTelemetry).
🧮 Code Interpreter Ejecuta código en un sandbox aislado, a pedido del agente.
🌐 Browser Navegación web headless y segura para tareas en la web.
USADO 🛡️ Guardrails Filtra temas y datos sensibles, en entrada y salida. El paso 4.1.
Hoy vamos a usar TRES bloques (los marcados): Runtime para hostear el agente, Memory para que recuerde la conversación, y Guardrails para filtrar datos sensibles. Y AgentCore tiene más, todos modulares, que se enchufan sin reescribir el agente: ¿conectar la API interna del cliente como tools? Gateway. ¿Que actúe en nombre del usuario con sus permisos? Identity. ¿Trazas y debugging? Observability. ¿Ejecutar código o navegar la web? Code Interpreter y Browser. Mensaje: empezás mínimo y escalás capacidades — justo el tipo de arquitectura que armamos en Craftech.
🧠 AgentCore Memory · corto plazo

Que recuerde la conversación

El historial de la charla, guardado por sesión. En cada turno el agente repite este ciclo — sin que el código lo maneje:

📥Cargar últimos turnos de la sesión
→
💬Responder con ese contexto
→
💾Guardar el turno nuevo
↺ se repite en cada mensaje
¿qué se decidió del deploy?
El deploy de prod quedó para el viernes 18hs.
¿y eso cuándo era?
Te referís al deploy: quedó para el viernes 18hs. ✅
Sin memoria, "eso" no significa nada. Con corto plazo, el agente sigue el hilo.

También hay largo plazo: extrae hechos y preferencias que persisten entre sesiones.

El agente sin memoria arranca de cero cada vez. Corto plazo = el historial de la charla por sesión: lo cargamos antes de responder y guardamos el turno nuevo (el ciclo de la izquierda). Por eso, mirá el chat de la derecha, podés decir "y eso cuándo era" y entiende que hablás del deploy. Y hay largo plazo si querés que recuerde cosas entre conversaciones distintas.

Hands-on · Paso 4

El agente que razona

Antes hacíamos retrieve a mano. Ahora un agente (Strands):

  • recibe la pregunta
  • decide usar la tool ask_kb
  • le pasa el contexto a Claude
  • Claude redacta la respuesta + citas

Dos tools: ask_kb y ingest_slack.
Por ahora corre local — sin desplegar nada.

@tool
def ask_kb(question: str) -> str:
    """Responde sobre el Slack indexado, con citas."""
    ctx, cites = retrieve(kb_id, question)
    return f"{ctx}\n\n[Fuentes: {cites}]"

agent = Agent(
    model=BedrockModel("claude-sonnet-4-6"),
    tools=[ask_kb, ingest_slack],
    system_prompt="Respondé sobre el Slack del equipo, "
                  "siempre con las fuentes.",
)

agent("¿cuándo es el deploy de prod?")
python -m s4_agent.main
Line-highlights: (1) la tool ask_kb es solo una función decorada; (2) el agente = modelo + tools + prompt; (3) lo invocás como una función. Correr en vivo y mostrar el "Tool #1: ask_kb" (el agente DECIDIENDO usar la herramienta) → ese es el momento "agente", no solo RAG.
Cómo razona · el agente

El agente decide y orquesta

Recibe la pregunta → decide usar ask_kb → busca en el Knowledge Base → le pasa el contexto a Claude → arma la respuesta con citas.

Seguí la señal: entra la pregunta al Agente; el agente DECIDE llamar a la tool ask_kb; esa tool va al Knowledge Base y trae los vecinos más cercanos (la señal se vuelve teal = el contexto recuperado); vuelve al agente, que se lo pasa a Claude; y Claude redacta la respuesta final con las citas. El punto: el agente no sigue un script fijo — DECIDE qué herramienta usar. Hoy es ask_kb; mañana podrían ser varias (buscar en Jira, postear, etc.).
Hands-on · Paso 5

Llevarlo a producción

El agente local pasa a AgentCore Runtime:

  • hosting serverless, sesión aislada
  • observabilidad incluida
  • sin contenedores ni servidores

Con el CLI agentcore: configure arma la config, deploy publica.

direct_code_deploy → no necesita Docker local.

# 1) configurar el runtime
agentcore configure \
    -e agent/agent.py \
    -er $RUNTIME_ROLE

# 2) desplegar con sus variables
agentcore deploy \
    --env KB_ID=$KB_ID \
    --env MODEL_ID=claude-sonnet-4-6
python -m s5_deploy_runtime.main --run
El salto: de "corre en mi máquina" a "serverless en AWS". Recalcar direct_code_deploy (sin Docker) y que el deploy tarda 1-2 min empaquetando deps. Tras desplegar: agentcore invoke '{"prompt":"..."}' para probar. Acá se puede mostrar la consola de Observability con la traza del agente (momento wow).
